# Knowledge Vault - Personal Knowledge Management System

A single-binary, serverless-ready Personal Knowledge Vault system that transforms book references into interconnected knowledge graphs using AI-powered analysis.

## 🎯 Vision

Build a Personal Knowledge Vault — a unified binary application that:
- Ingests book references via Open Library API
- Processes data asynchronously with intelligent concept extraction
- Stores and visualizes knowledge as interconnected graphs
- Runs as a single, zero-dependency binary on resource-constrained environments

- ✅ **Single Binary**: Zero external service dependencies
- ✅ **Embedded Database**: SurrealKV for embedded SurrealDB

- ✅ **Hexagonal Architecture**: Clear domain isolation via Traits
- ✅ **ARM64 Optimized**: Oracle Cloud Free Tier ready

## 📊 Domain Model

### Entities (Nodes)

```
Work
├── id: string (Primary)
├── title: string
├── author: string
├── isbn: string
├── edition: string
└── metadata: object

Insight
├── id: string (Primary)
├── work_id: string (FK)
├── summary: string (Generated by Gemini)
├── key_themes: [string]
└── generated_at: datetime

Concept
├── id: string (Primary)
├── name: string
├── category: enum (Theory | Practice | Entity | Theme)
├── description: string
└── embedding: vector (Optional)
```

### Relationships (Edges)

```
Work --[interpreta]--> Insight
  └── One work generates multiple insights

Insight --[menciona]--> Concept
  └── Each insight references multiple concepts

Concept --[relacionado_a]--> Concept
  └── Concepts form interconnected knowledge graph
  └── Bidirectional relationship with strength weights
```

### SurrealQL Schema

```surrealql
-- Works table

DEFINE FIELD id ON TABLE work TYPE string;
DEFINE FIELD title ON TABLE work TYPE string;

DEFINE FIELD metadata ON TABLE work TYPE object;

-- Insights table
CREATE TABLE insight SCHEMAFULL;
DEFINE FIELD id ON TABLE insight TYPE string;
DEFINE FIELD work_id ON TABLE insight TYPE record<work>;
DEFINE FIELD summary ON TABLE insight TYPE string;
DEFINE FIELD key_themes ON TABLE insight TYPE array;
DEFINE FIELD generated_at ON TABLE insight TYPE datetime;

-- Concepts table
CREATE TABLE concept SCHEMAFULL;
DEFINE FIELD id ON TABLE concept TYPE string;
DEFINE FIELD name ON TABLE concept TYPE string UNIQUE;
DEFINE FIELD category ON TABLE concept TYPE string;
DEFINE FIELD description ON TABLE concept TYPE string;
DEFINE FIELD embedding ON TABLE concept TYPE array;

-- Relationships
CREATE TABLE interpreta SCHEMAFULL;
DEFINE FIELD in ON TABLE interpreta TYPE record<work>;
DEFINE FIELD out ON TABLE interpreta TYPE record<insight>;

CREATE TABLE menciona SCHEMAFULL;
DEFINE FIELD in ON TABLE menciona TYPE record<insight>;
DEFINE FIELD out ON TABLE menciona TYPE record<concept>;

CREATE TABLE relacionado_a SCHEMAFULL;
DEFINE FIELD in ON TABLE relacionado_a TYPE record<concept>;
DEFINE FIELD out ON TABLE relacionado_a TYPE record<concept>;
DEFINE FIELD strength ON TABLE relacionado_a TYPE number DEFAULT 1.0;
DEFINE FIELD context ON TABLE relacionado_a TYPE string;
```

## 🔄 Analysis Flow

### 1️⃣ Ingestion Phase
```
User Input (ISBN/Title)
    ↓
Axum HTTP Handler validates Request
    ↓
Publishes WorkIngestEvent to NATS
```

### 2️⃣ Async Processing Phase
```
NATS Worker consumes WorkIngestEvent
    ↓
Enriches data via Open Library API
    ↓
Sends enriched context to Gemini API
    ↓
Extracts concepts & knowledge relationships
```

### 3️⃣ Persistence Phase
```
Worker queries Open Library API
    ├── Retrieves book metadata
    ├── Enriches Work node
    └── Creates Insight node

Gemini API processes content
    ├── Extracts key concepts
    ├── Identifies relationships
    └── Generates executive summary

SurrealDB persists graph
    ├── Writes Work/Insight/Concept nodes
    ├── Creates relationship edges
    └── Updates concept connections
```

### 4️⃣ Notification Phase
```
Worker completion event
    ↓
Frontend receives via WebSocket/Signal
    ↓
UI reflects Work as processed & graph updated
```

}

// Domain Service
pub struct KnowledgeOrchestrator {
    discovery: Arc<dyn DiscoveryService>,
    // ... other dependencies

### Output Ports (Traits)

```rust
// Repository Pattern
pub trait WorkRepository {
    async fn save_work(&self, work: Work) -> Result<WorkId, RepositoryError>;
    async fn find_by_isbn(&self, isbn: &str) -> Result<Option<Work>, RepositoryError>;
}

pub trait ConceptRepository {
    async fn save_concept(&self, concept: Concept) -> Result<ConceptId, RepositoryError>;
    async fn find_related(&self, concept_id: ConceptId) 
        -> Result<Vec<Concept>, RepositoryError>;
}

// Intelligence Port
pub trait IntelligencePort {
    async fn extract_concepts(&self, context: &str) 
        -> Result<ConceptSet, IntelligenceError>;
    async fn generate_insight(&self, work: &Work) 
        -> Result<Insight, IntelligenceError>;
}

// Message Port
pub trait MessagePort {
    async fn publish(&self, event: DomainEvent) -> Result<(), MessagingError>;
    async fn subscribe(&self, topic: &str, handler: Box<dyn EventHandler>) 
        -> Result<Subscription, MessagingError>;
}

// External Data Port
pub trait ExternalDataPort {
    async fn lookup_book(&self, isbn: &str) -> Result<BookMetadata, ExternalError>;
    async fn search_books(&self, query: &str) -> Result<Vec<BookMetadata>, ExternalError>;
}

// Authentication Port
pub trait AuthPort {
    async fn validate_token(&self, token: &str) -> Result<Claims, AuthError>;
    async fn validate_pat(&self, pat: &str) -> Result<PatClaims, AuthError>;
}
```

### Resilience Strategy (NATS)

```rust
pub trait RetryPolicy {
    fn should_retry(&self, attempt: u32, error: &Error) -> bool;
    fn next_delay(&self, attempt: u32) -> Duration;
}

// Exponential backoff with jitter for external API failures
pub struct ExponentialBackoff {
    base_delay: Duration,
    max_retries: u32,
    max_delay: Duration,
}

// NATS publish with automatic retries
pub async fn publish_with_retry(
    event: DomainEvent,
    policy: &dyn RetryPolicy,
) -> Result<(), Error> {
    let mut attempt = 0;
    loop {
        match nats_client.publish(&event).await {
            Ok(_) => return Ok(()),
            Err(e) if policy.should_retry(attempt, &e) => {
                tokio::time::sleep(policy.next_delay(attempt)).await;
                attempt += 1;
            }
            Err(e) => return Err(e),
        }
    }
}
```

## 🔐 Authentication

### JWT Configuration
- **Algorithm**: HS256
- **Payload**: User ID, IAT, EXP, Scopes
- **Storage**: Httponly Cookie (Frontend managed by Leptos)

### Personal Access Tokens (PAT)
- **Format**: `ens_<random_128bit_base64>`
- **Storage**: Hashed in SurrealDB with rotation metadata
- **Usage**: Service-to-service, batch operations

```rust
pub struct PatRequest {
    name: String,
    expiry: Option<DateTime<Utc>>,
}

pub struct Pat {
    id: String,
    prefix: String, // "ens_"
    hash: String,   // SHA256(token)
    created_at: DateTime<Utc>,
    expires_at: Option<DateTime<Utc>>,
}
```

### Setup & Authentication Endpoints

**Check first-run status**
```
GET /api/setup
Response: 200 {first_run: bool}
```

> Note: `GET /setup` (no `/api/` prefix) serves the browser HTML setup page and redirects to `/login` if already configured.

**Create admin account (first-run only)**
```
POST /api/setup
Content-Type: application/json

{
  "email": "admin@example.com",
  "password": "your-12+-character-password"
}

Response: 201 {user_id: "..."}
Error: 409 (already configured), 422 (validation error)
```

> Note: `POST /setup` (no `/api/` prefix) accepts `application/x-www-form-urlencoded` and is used by the browser form.

**Login with email and password**
```
POST /auth/login
Content-Type: application/json

{
  "email": "admin@example.com",
  "password": "your-12+-character-password"
}

Response: 200
Set-Cookie: session=<JWT>; HttpOnly; Secure; SameSite=Strict
{
  "token": "<JWT token>"
}

Error: 401 (invalid credentials), 429 (rate limited)
```

#### Login Details
- **Rate Limiting**: Maximum 3 failed attempts per email in a 5-minute window
- **Rate Limit Response**: `429 Too Many Requests` with `retry_after_seconds` field
- **Token Expiry**: 24 hours from issuance
- **Cookie Attributes**:
  - `HttpOnly`: Prevents JavaScript access (mitigates XSS)
  - `Secure`: Only transmitted over HTTPS
  - `SameSite=Strict`: CSRF protection
  - `Path=/`: Cookie sent for all paths
- **Password Requirements**: Minimum 12 characters
- **Password Hashing**: Argon2id (OWASP 2026 parameters: m=65536, t=3, p=1)

### Security Headers
All responses include the following security headers:
- `X-Content-Type-Options: nosniff` — Prevents MIME type sniffing
- `X-Frame-Options: DENY` — Prevents clickjacking
- `Content-Security-Policy: default-src 'self'; script-src 'self' 'wasm-unsafe-eval'; style-src 'self' 'unsafe-inline'` — Restricts resource loading
- `Referrer-Policy: strict-origin-when-cross-origin` — Controls referrer information

## 🔍 Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **SurrealDB Embedded** | Zero external dependencies, ideal for single binary |
