## User Story

**As Beatriz**, I want to see the real-time processing status of my submitted books, so that I know when my knowledge graph is ready.

---

## Acceptance Criteria

```
Given I am authenticated (JWT or PAT)
When I call GET /api/works
Then the response is HTTP 200 with an array of works, each including:
  work_id, title, author, isbn, status, error_msg, created_at

Given I am not authenticated
When I call GET /api/works
Then the response is HTTP 401

Given I am authenticated and a valid work_id exists
When I call GET /api/works/{id}
Then the response is HTTP 200 with the work object

Given I am authenticated and work_id does not exist
When I call GET /api/works/{id}
Then the response is HTTP 404

Given I am not authenticated
When I call GET /api/works/{id}
Then the response is HTTP 401

Given I am authenticated
When I connect to GET /ws via WebSocket upgrade
Then the connection is upgraded and the client receives real-time JSON messages
  of the form: {"type": "work_status", "work_id": "<uuid>", "status": "<status>"}
  whenever any work's status changes

Given I am not authenticated
When I attempt to connect to GET /ws
Then the response is HTTP 401 (no WebSocket upgrade)
```

## Technical Context

### Endpoints

```
GET /api/works
Authorization: Bearer <jwt_or_pat>
Query: ?limit=50&offset=0 (defaults: limit=50, max=200)
Response 200: [{work_id, title, author, isbn, status, error_msg, created_at}]
Response 401: {"error": "unauthorized", "message": "..."}

GET /api/works/{id}
Authorization: Bearer <jwt_or_pat>
Response 200: {work_id, title, author, isbn, status, error_msg, created_at}
Response 401: {"error": "unauthorized", "message": "..."}
Response 404: {"error": "not_found", "message": "Work not found"}

GET /ws
Authorization: Bearer <jwt_or_pat>
Protocol: WebSocket upgrade
Server push format: {"type": "work_status", "work_id": "<uuid>", "status": "<status>"}
Response 401: {"error": "unauthorized", "message": "..."} (if auth fails, no upgrade)
```

`update_work_status` is needed by S02-04 (NATS Worker) and S02-06 (Manual Retry) but its interface is defined here.

### WebSocket broadcaster

New file `src/web/ws_broadcaster.rs`:

```
WsBroadcaster:
  - Uses tokio::sync::broadcast channel (capacity 256)
  - broadcast(work_id, status): serialises {"type":"work_status","work_id":...,"status":...} and sends
  - subscribe(): returns Receiver<String> for a new WebSocket client
```

The broadcaster is stored as `Arc<WsBroadcaster>` in `AppState`. S02-04 (NATS Worker) and the WebSocket handler both access it via the shared state.

### Handler locations

- `src/web/handlers/works.rs` — add `get_works` and `get_work_by_id`
- `src/web/handlers/websocket.rs` — new file, WebSocket upgrade handler

### Auth middleware

All three endpoints are protected by the existing JWT/PAT middleware. The `AuthenticatedUser` extractor rejects unauthenticated requests before the handler runs (or before the WebSocket upgrade).

## Implementation Notes

### SurrealDB queries

`list_works`:
```surrealql
SELECT * FROM work ORDER BY created_at DESC LIMIT $limit START $offset
```

`get_work_by_id`:
```rust
self.db.select(("work", id)).await
```

`update_work_status`:
```surrealql
UPDATE work SET status = $status, error_msg = $error_msg, updated_at = time::now() WHERE id = work:{id}
```

### WebSocket lifecycle

1. Client connects to GET /ws; auth middleware validates JWT/PAT
2. Handler upgrades to WebSocket, creates a broadcast receiver
3. Background task forwards broadcaster messages to client via `socket.send()`
4. On client disconnect, task exits and receiver is dropped (no cleanup needed)

### Logging

On WebSocket connect: `INFO user_id=<id> "WebSocket connected"`
On WebSocket disconnect: `INFO user_id=<id> "WebSocket disconnected"`

## Testing

- broadcast sends message to all active subscribers
- new subscriber after broadcast does not receive stale messages
