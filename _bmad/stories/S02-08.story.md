## User Story

**As Beatriz**, I want concept nodes and their relationships to be persisted correctly in the graph database, so that the knowledge graph accurately reflects the book's insights without duplicates.

---

## Acceptance Criteria

```
Given a GeminiResponse containing concepts
When the worker persists the response
Then each concept is upserted by normalized name (lowercase + trimmed)
And duplicate concepts from different books are deduplicated — same normalized name returns the same record
And a new Insight record is created with summary, key_points, and raw_gemini_response
And an interpreta edge is created from the work to the insight
And a menciona edge is created from the insight to each concept, carrying relevance_weight
And a relacionado_a edge is created for each related_concept pair, carrying relation_type and strength

Given a concept with the same normalized name already exists
When the worker calls upsert_and_link again for the same concept
Then the existing concept record is returned without creating a duplicate

Given a relevance_weight is omitted by Gemini for a menciona edge
When persisting
Then the edge is created with a default weight of 0.5

Given a strength is omitted by Gemini for a relacionado_a edge
When persisting
Then the edge is created with a default strength of 0.5

Given a relation_type is omitted by Gemini for a relacionado_a edge
When persisting
Then the edge is created with a default relation_type of "related"

Given a work was already processed (interpreta edge already exists)
When create_insight is called again for the same work
Then the existing insight ID is returned without creating a duplicate
```

## Technical Context

### Domain types

**`src/domain/concept.rs`** — `Concept`, `normalize_concept_name`

**`src/domain/insight.rs`** — `Insight`, `GeminiResponse`, `ExtractedConcept`, `RelatedConceptRef`
- `ExtractedConcept.relevance_weight: f64` — mandatory, caller must default to 0.5 if Gemini omits
- `RelatedConceptRef.relation_type: String` — `#[serde(default = "default_relation_type")]` → "related"
- `RelatedConceptRef.strength: f64` — `#[serde(default = "default_strength")]` → 0.5

### Port traits (`src/ports/repository.rs`)

```rust
pub trait InsightRepo {
    async fn create_insight(
        &self,
        work_id: &str,
        summary: &str,
        key_points: Vec<String>,
        raw_json: &str,
    ) -> Result<String, RepoError>;  // returns insight UUID
}

pub trait ConceptRepo {
    async fn upsert_and_link(
        &self,
        work_id: &str,
        insight_id: &str,
        concepts: Vec<ExtractedConcept>,
    ) -> Result<(), RepoError>;
}
```

### SurrealDB schema

Tables: `concept` (UNIQUE index on `name`), `insight`, `interpreta` (IN work OUT insight),
`menciona` (IN insight OUT concept, `relevance_weight`), `relacionado_a` (IN concept OUT concept,
`relation_type + strength`).

### Deduplication logic

`upsert_and_link` normalizes each concept name (`normalize_concept_name`), looks up by normalized
name, and only creates a new record if not found. First-write wins. `create_insight` is idempotent:
if the work already has an `interpreta` edge, it returns the existing insight ID.

### Adapter files

- `src/adapters/surreal/concept_repo.rs` — `SurrealConceptRepo` implementing `ConceptRepo`
- `src/adapters/surreal/insight_repo.rs` — `SurrealInsightRepo` implementing `InsightRepo`
