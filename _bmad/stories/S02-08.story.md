## User Story

**As Beatriz**, I want concept nodes and their relationships to be persisted correctly in the graph database, so that the knowledge graph accurately reflects the book's insights without duplicates.

---

## Acceptance Criteria

```
Given a GeminiResponse containing concepts
When the worker persists the response
Then each concept is upserted by normalized name (lowercase + trimmed)
And duplicate concepts from different books are deduplicated — same normalized name returns the same record
And a new Insight record is created with summary, key_points, and raw_gemini_response
And an interpreta edge is created from the work to the insight
And a menciona edge is created from the insight to each concept, carrying relevance_weight
And a relacionado_a edge is created for each related_concept pair, carrying relation_type and strength

Given a concept with the same normalized name already exists
When the worker calls upsert_concept again
Then the existing concept record is returned without creating a duplicate
And the concept's display_name and description are NOT overwritten (first-write wins)

Given two concepts A and B with a relacionado_a relationship
When the edges are created
Then a relacionado_a edge from A to B exists with the correct relation_type and strength

Given a relevance_weight is omitted by Gemini for a menciona edge
When persisting
Then the edge is created with a default weight of 0.5

Given a strength is omitted by Gemini for a relacionado_a edge
When persisting
Then the edge is created with a default strength of 0.5

Given a relation_type is omitted by Gemini for a relacionado_a edge
When persisting
Then the edge is created with a default relation_type of "related"
```

### New port traits

**`src/ports/repository.rs` additions**
```rust
pub trait ConceptRepo {
    async fn upsert(&self, display_name: &str, description: &str, domain: &str) -> Result<Concept, RepoError>;
    async fn find_by_name(&self, normalized_name: &str) -> Result<Option<Concept>, RepoError>;
    async fn create_menciona_edge(&self, insight_id: &str, concept_id: &str, weight: f64) -> Result<(), RepoError>;
    async fn create_relacionado_a_edge(&self, from_id: &str, to_id: &str, relation_type: &str, strength: f64) -> Result<(), RepoError>;

pub trait InsightRepo {
    async fn create(&self, summary: &str, key_points: Vec<String>, raw_response: &str) -> Result<Insight, RepoError>;
    async fn create_interpreta_edge(&self, work_id: &str, insight_id: &str) -> Result<(), RepoError>;
}
```

### Schema additions (schema.rs)

```surrealql
DEFINE TABLE IF NOT EXISTS concept SCHEMAFULL;
DEFINE FIELD IF NOT EXISTS name         ON concept TYPE string;
DEFINE FIELD IF NOT EXISTS display_name ON concept TYPE string;
DEFINE FIELD IF NOT EXISTS description  ON concept TYPE string;
DEFINE FIELD IF NOT EXISTS domain       ON concept TYPE string;
DEFINE FIELD IF NOT EXISTS created_at   ON concept TYPE datetime DEFAULT time::now();
DEFINE INDEX IF NOT EXISTS concept_name ON concept COLUMNS name UNIQUE;

DEFINE TABLE IF NOT EXISTS insight SCHEMAFULL;
DEFINE FIELD IF NOT EXISTS summary             ON insight TYPE string;
DEFINE FIELD IF NOT EXISTS key_points          ON insight TYPE array<string>;
DEFINE FIELD IF NOT EXISTS raw_gemini_response ON insight TYPE string;
DEFINE FIELD IF NOT EXISTS created_at          ON insight TYPE datetime DEFAULT time::now();

DEFINE TABLE IF NOT EXISTS interpreta SCHEMAFULL TYPE RELATION IN work OUT insight;

DEFINE TABLE IF NOT EXISTS menciona SCHEMAFULL TYPE RELATION IN insight OUT concept;
DEFINE FIELD IF NOT EXISTS relevance_weight ON menciona TYPE float;

DEFINE TABLE IF NOT EXISTS relacionado_a SCHEMAFULL TYPE RELATION IN concept OUT concept;
DEFINE FIELD IF NOT EXISTS relation_type ON relacionado_a TYPE string;
DEFINE FIELD IF NOT EXISTS strength      ON relacionado_a TYPE float;
```

### Deduplication logic

`upsert_concept` normalizes the name, checks for an existing record with `SELECT * FROM concept WHERE name = $name LIMIT 1`, and returns it if found. Only if no record exists does it insert a new one. First-write wins — existing display_name and description are preserved.
